# AGENTS.md

## 项目定位

本仓库用于存放课堂教学使用的小应用，面向学生课堂演示与互动练习。

硬性约束如下：

- 所有项目均为纯静态页面应用，优先使用原生 `HTML`、`CSS`、`JavaScript`。
- 所有应用都通过 GitHub Pages 部署。
- 每一个应用必须对应仓库根目录下的一个独立目录，目录名即访问路径的一部分。
- 每次新增、修改、下线应用时，必须同步更新 `README.md` 中的应用清单与功能说明。
- AI 完成任务后，默认流程为：整理变更、合并到 `main` 分支、推送到远端；除非用户明确要求不要执行。

## 核心原则

- 保持简洁，遵循 KISS，拒绝过度工程化。
- 从第一性原理出发理解需求，先确认真实目标，再开始实现。
- 以事实为准，不确定时先调研仓库和上下文，避免凭空假设。
- 优先交付可直接运行、可直接访问、可直接教学使用的结果。

## 沟通与输出要求

- 所有回复、分析、计划、任务清单均使用中文。
- 固定指令：`Implementation Plan, Task List and Thought in Chinese`
- 在设计或编码前，必须先完成前期调研，明确当前仓库状态、现有目录结构和任务边界。
- 严格遵循工作顺序：`构思方案 -> 提请审核 -> 分解为具体任务 -> 实施 -> 验证 -> 更新文档 -> 合并 main -> 推送远端`

## 目录规范

仓库根目录推荐保持如下结构：

```text
.
├── AGENTS.md
├── README.md
├── app-name-a/
│   ├── index.html
│   ├── style.css
│   ├── script.js
│   └── assets/
├── app-name-b/
│   └── index.html
└── shared/
```

具体规则：

- 每个应用使用仓库根目录下的独立目录，例如 `timer-game/`、`flash-card/`。
- 每个应用目录至少包含一个可直接访问的 `index.html`。
- 应用目录名使用短横线命名，简洁、语义清晰、便于形成稳定 URL。
- 默认不要引入构建工具、打包器、Node 服务端或复杂工程脚手架。
- 只有在多个应用确实需要复用时，才允许增加 `shared/` 目录；不要过早抽象。
- 静态资源应尽量放在应用自身目录内，避免不同应用互相耦合。

## GitHub Pages 约束

- GitHub Pages 采用静态托管思路设计，不依赖服务端能力。
- 所有链接、脚本、样式、图片路径都应优先使用相对路径，确保在仓库子路径下正常访问。
- 默认访问形式为：
  - 仓库首页：`https://<github-username>.github.io/xiaoke/`
  - 单个应用：`https://<github-username>.github.io/xiaoke/<app-directory>/`
- 若仓库启用了自定义域名，也必须保证目录结构仍能按路径独立访问。

## 应用开发要求

- 优先保证打开即用，不需要额外安装依赖。
- 优先保证课堂场景下的稳定性和可理解性，界面清晰、文字直白、交互明确。
- 默认兼容桌面端与常见手机浏览器。
- 若涉及音频、图片、字体等外部资源，优先本地化或使用稳定来源，并确认 GitHub Pages 可正常加载。
- 若实现较复杂，需在应用目录内添加简短说明或清晰注释，方便后续维护。

## README 维护要求

每次对应用进行任何变更时，都必须同步维护根目录 `README.md`，至少包括：

- 应用目录名
- 应用名称
- 功能说明
- GitHub Pages 访问路径
- 当前状态（可用、开发中、已下线等）

若 `README.md` 未更新，则视为任务未完成。

## Git 工作流要求

- 开始前先确认当前分支状态，避免覆盖用户已有修改。
- 若任务涉及实现，先给出方案，再实施。
- 完成后至少进行一次基本验证，例如检查文件结构、关键路径、链接和文档内容。
- 默认交付流程：
  - 确认变更内容
  - 合并到 `main`
  - 推送到 `origin/main`
- 非用户明确要求，不要改写历史，不要使用破坏性 Git 命令。

## 任务完成检查清单

提交结果前，至少确认以下事项：

- 应用是否位于独立目录下
- 应用是否可通过对应目录路径访问
- 资源路径是否适配 GitHub Pages 子路径部署
- `README.md` 是否已登记该应用及功能
- 是否遵循纯静态、简单可维护的原则
- 是否已准备合并 `main` 并推送
